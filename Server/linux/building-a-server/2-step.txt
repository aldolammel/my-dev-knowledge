

DEBIAN/UBUNTU > BUILDING UP A WEB-SERVER: THE BASIC


    PRE.1) Assuming you have finished the previous step!
    
    
    PRE.2) You're in the VPS terminal!


    1) (Optional) Customizing the ssh_config file:
    
        1.1) Server-side (VPS):
        
            1.1.PRE.1) Assuming you are in the VPS terminal:
                /Server/connecting-via-ssh.txt
            
            1.1.PRE.2) Open to the file:
                $ sudo nano /etc/ssh/ssh_config
        
            1.1.PRE.3) Use this command to find specific directives:
                Ctrl + W
            
            1.1.1) Edit them:
                # Seconds between keepalive probes sent to the client:
                    ClientAliveInterval 60
                # How many unanswered probes before the server drops the connection:
                    ClientAliveCountMax 20
            
                TCPKeepAlive yes
                # this feat. enables the periodic transmission of KeepAlive msgs to detect if a connection is still active
        
            1.1.2) Save (ctrl+s) and exit (ctrl+x);
        
            1.1.3) Restart the SSH service on VPS:
                $ sudo systemctl restart sshd
                
                # Not need to reboot or close the terminal!
        
        
        1.2) Client-side (your local machine):
        
            1.2.PRE) Assuming you are in the local terminal!
    
            1.2.1) Go to the file:
            
                # Affecting only your local user (RECOMMENDED):
                    /home/<your_user>/.ssh/config
                    $ nano /home/<your_user>/config    <- It's a new file!
                    
                # Or affecting all local users:
                    /etc/ssh/ssh_config
                    $ sudo nano /etc/ssh/ssh_config     <- Existing file!
            
            1.2.2) Add these lines and save it:
                    
                >> For all cases (RECOMMENDED):
                    Host *
                        ServerAliveInterval 60
                        ServerAliveCountMax 20
                        
                >> Or for specific host/VPS:
                    Host <your-vps-name>
                        HostName <your.vps.ip.or.domain>
                        User <youruser>
                        ServerAliveInterval 60
                        ServerAliveCountMax 20
                        
                INFO:
                    For client-side, no need to restart any service once the local SSH 'config' is read automatically for every new SSH connection called on local terminal.
        
        
    2) (optional) Configuring the firewall:
    
        2.PRE.1) You should be in the VPS terminal now!
        
        2.PRE.2) Install one: using the UFW
            $ sudo apt install ufw
        
        2.1) Allowing basic permissions:
            By default, all incoming connections to our server are denied, even the SSH connection on port 22. If we log out to the server, we can't connect anymore. Let's allow SSH connection on port 22 with the command below:
        
            # Allow ssh connection
                $ ufw allow ssh

        2.2) Activating the firewall:
            $ ufw enable               # View all rules defined
            $ ufw status verbose

            INFO:
                We don't add sudo because we are connected with the root user. If it is not your case, consider adding the "sudo" keyword before the command.
            
        2.3) Test the SSH connection after:
            Log out, then log in again to ensure you can still connect to the server!
        
        
    3) (Optional - HIGHLY RECOMMENDED) For security reasons, change SSH Port:
    
        By default, the SSH port is 22. Changing this to make your server a little more secure is a good idea. Here is the process to change the port to 4928.
    
        3.1) Open the SSH config's file:
            $ nano /etc/ssh/sshd_config
        
            Locate line #Port 22
            Remove the hashtag in front of and replace 22 by something like 4928
            Save the file and exit!
        
        3.2) Restart the SSH service: 
            $ /etc/init.d/ssh restart
    
        3.3) (If applicable) Now, we need to update the firewall rule to:
        
            3.3.1) Allow SSH connection on port:
                $ ufw allow <PORTNUMBERSELECTED>/tcp
    
            3.3.2) Delete rule for SSH connection on port 22:
                This is achieved in two steps where the first is to list the rule with a number assigned to each rule:
                $ ufw status numbered
            
                # Locate the number of the rule you want to delete, then type: 
                    $ ufw delete <rule_number>       # e.g: ufw delete 1
                
                # Refresh the index:
                    $ ufw status numbered
                
                # Repeat the same for "22/tcp (v6)" too!
                    $ ufw delete <rule_number>
                
                # Check if only what you need is listed:
                    $ ufw status numbered
            
        
    4) (Optional) For security reasons, create a <new_user> with a sudo privilege:
    
        Since then, we have used the root user to perform restricted actions. It is not a good decision because a bad manipulation can cause significant damage. It is better to create a user with lower privileges on the system and keep the user performing actions requiring the root user.
        
        4.1) The command below creates a <new_user> with sudo privilege:
            # create user admin with his home directory
                $ useradd -m -d /home/<new_user> <new_user>

            # Start the process to set a password for the new user
                $ passwd <new_user>
                    # now it will ask you for a new password ;)

            # add the user in the sudo users group:
                $ usermod -aG sudo <new_user>
            
            # Verify that:
                $ groups <new_user>

        4.2) Log out, and then connect with the <new_user>!
        
        4.3) (Optional - NOT RECOMMENDED) For security reasons, deactivating the root user access:
        
            CRITICAL INFO:
                If you block the root user of SSH connections (by dev terminal, for example), if a config to demand root user, you should use the terminal by the VPS provider web console, and there - SOMETIMES - Brazilian keyboards are not correctly accepted.             INFO:

            4.3.PRE.1) Only do it after to create a <new_user> user!!!
        
            4.3.PRE.2) Logged in VPS terminal using the <new_user>!
        
            4.3.1) Update the SSHD_CONFIG file:
                $ sudo nano /etc/ssh/sshd_config
                Change the text "PermitRootLogin yes" to "PermitRootLogin no". And then save it (ctrl+s) and exit (ctrl+x).
        
            4.3.2) Restart the ssh service:
                $ sudo systemctl restart ssh
            
            4.3.3) Test it:
                # Logout the VPS terminal:
                $ exit
                    # And test the SSH connection again but using the root! 
                    # You must see a message like "Access denied".
                    
            4.3.4) Reconnect with your non-root-user and have fun!
        
        
    5) Install a Web server:
    
        Our goal is to host our web app and make it accessible through the internet, but a server is just a computer like ours but without a GUI and up every time. By default, the server doesn't have this capability, and we need to install a Web server to make this possible. The most popular web servers are Apache, Nginx, and Microsoft IIS.
        
        5.PRE) We are going to install Nginx, so check if you have Apache running on the server:
        
            # If so, stop it to avoid Nginx headaches:
                $ apache2 -v
                # If nothing was found, this is good, but if there's Apache installed, remove it:
                    $ sudo apt autoremove
                    $ sudo apt remove apache2.*
        
        5.1) Installing the webserver:
        
            # After that, keep going:
                $ sudo apt install nginx -y
            
            # (If applicable) Gives the Webserver the firewall permission:
                $ sudo ufw allow 'Nginx Full'
                
                INFO:
                    The command sudo ufw allow 'Nginx Full' open ports 80 and 443 for HTTP and HTTPS, respectively, to allow incoming traffic from the web. So it's not necessary to have allowed "Nginx HTTP" and "Nginx HTTPS". Only "Nginx Full" must be allowed.
            
            # Check the webserver status:
                $ sudo systemctl status nginx
            

    6) (Optional) Security reasons, connect to the server with a private key:
    
        IMPORTANT:
            Keys are exclusively used by each system's user. If you are the admin user, maybe in the future you will need to create one for the root user.

        We connect to our server with a username and password, but connecting with an RSA private key is also possible. Let's see how to create that key and connect to the server using that:

        >> RSA:
            An SSH RSA key is considered highly secure as it has typically larger key sizes, often 2048 or 4096 bits. It's also more compatible with older OS.
        
        >> Ed25519:
            A more modern algorithm with a smaller standard key size of 256 bits. It is just as secure and efficient as an RSA key due to its strong cryptographic properties. The compatibility is lower, but newer operating systems support it.

        6.1) Once connected to the VPS, generate the key:
        
            >> Generate an RSA key pair:
                $ ssh-keygen -t rsa
            
            >> Or this one:
                $ ssh-keygen -t ed25519
        
        
        6.2) After that, the key generator should ask you to choose a location to store the keys. Just press ENTER to confirm the key location (e.g. /home/local_username/.ssh/id_rsa)!
        
        6.3) The key generator window will ask you to create an SSH key passphrase to access your private key. Set one as strong as possible;
        
            Tip:
                Use Norton Generated Password with 20 char length and special chars:
                    https://us.norton.com/feature/password-generator
                    
            This password, you will use, for example, when you GIT CLONE the repo!
        
            # Also copy the entire output (if applicable) and save it somewhere safely!
        
        6.4) Log out of the server and re-connect normally!
        
        6.5) (Optional) It's advised to create a Public Key for the root user in cases where only they can set a repo in places where the admin has no write permission (as "/var/www/" for example). If you decide to do it, go to the VPS provider and use its terminal directly from there to configure a Public Key exactly like you just have done here, but for the root user!
        
    7) Prepare the Server to receive clones of GitHub repositories:
    
        7.PRE.1) Back again logged, find your user root-folder:
        
        7.PRE.2) Check where you are in the VPS terminal:
            $ pwd
            
        7.1) If you are ROOT, the folder is /root/, or if you are another sudo user, the folder is /home/<your_user>/:
        
        # In your user folder, get in the hidden folder 'ssh':
            $ cd .ssh
        
        7.2) Let's copy our user Public Key:
            # list the current folder content to check if you can see your public key (e.g. id_rsa.pub):
                $ ls
            
            # Copy the entire code on the terminal and let's go to GitHub:
                $ cat id_rsa.pub

        7.3) On GitHub, go to:
            >> Settings >> SSH and GPG Keys >> New SSH Key;
        
            Title example = <Hosting name>_<server_OS>_<username>
                e.g. locaweb_ubuntu_admin
            
            And then "Add SSH key"!    
            Now you are prepared to clone private GitHub repos into the server.
