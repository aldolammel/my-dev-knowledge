

DJANGO FORMS VALIDATIONS: FOR APP LEVEL (FINAL USER'S FRONT-END)

    IMPORTANT:
        All validation for db-level (only written through the models.py file) Django manages to bring them to app-level (customizable by forms.py), automatically, even if the project is not using Django Template as front-end solution. If youâ€™re not using Django Template in your front-end stack, you may need to manually replicate validations in your chosen front-end framework (e.g., React, Vue). However, back-end validations in models.py always applies.

    Considering "Field-level" means any field from a Model-class and ModelForm-class, there are these approaches to set each validation logic:

        >> For app validations using native Django Template solution:

            A. In models.py, model-field-level validation (using "validators");
            B. In models.py, model-level validation (using "clean()");
            C. In forms.py, model-form-field-level validation (using "clean_<field>()");
            D. In forms.py, model-form-level validation (using "clean()");
            
        >> For app validations that's NOT using native Django Template solution:

            >> Cases 'A' and 'B' certainly 'coz you still need validations for CMS-level once you wanna protect the data coming through admin inputs. So, validator functions and clean() method defined directly in models.py classes are propagated also to CMS-level, automatically.

            >> Cases 'C' and 'D' you may need when you customize CMS forms (detail-views) tied with ModelForms from forms.py once admin.py doesn't accept directly the clean() use.

    All that said, keep in mind:


    FIELD-LEVEL - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - 

        Using validators:
            1. Recommended to used only for model-class' fields;
            2. Used as argument of model fields;
            3. You can use built-in validators or customized ones;
            4. They run when the full_clean() is explicity called through the save() method;
            5. They NEVER validate in db-level;
            6. They impact in app-level, forms-level, and model-level.

            >> More about:
                ./models-validators-built-in.txt
                ./models-validators-customized.py
        
        Using clean_<field>():
            1. Used only for model-form-class' fields;
            2. Used as method of model form fields;
            3. You must create every field validation you need;
            4. It runs automatically or when it's triggered by .is_valid() in views.py;
            5. It NEVER validate in db-level and even in model-level;
            6. It impacts in app-level and forms-level;

            >> More about:
                ./clean-differences-between-model-and-form.txt


    CROSS-FIELD-LEVEL - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - 

        Using clean():
            1. Used in model-class and model-form-class;
            2. Used as method to check 2 or more fields relation (cross) or complex fields;
            3. You must create every validation you need;
            
            Only if use in model-classes:
                4. It runs when the full_clean() is explicity called through the save() method;
                5. They NEVER validate in db-level;
                6. They impact in app-level, forms-level, and model-level.

            Only if use in model-form-classes:
                4. It runs automatically or when it's triggered by .is_valid() in views.py;
                5. It NEVER validate in db-level and even in model-level;
                6. It impacts in app-level and forms-level;
            
            >> More about:
                ./clean-differences-between-model-and-form.txt


    IMPORTANT:
        Avoid to set clean() in forms.py file if you've already done that in models.py once it propagates its clean() to forms.py too! Otherwise, if you need to know what the user's doing, for example, in a form field before they save something on db, you definitely need to apply clean() in forms.py:
            .../django/9-forms/form-validation-getting-data.py


    CRITICAL:
        If you bypass forms usage (e.g., calling Product.objects.create() directly), clean() method WON'T run automatically too.


    >> How to get data for custom validations in Model clean() method:
        ./validation-clean-getting-data.py
